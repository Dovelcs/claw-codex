# codex-server

`codex-server` provides bridge utilities for controlling Codex app-server
threads from remote clients, plus a fleet manager and OpenWrt-side patch
scripts used by the Feishu integration.

## Repository Layout

- `src/`: TypeScript bridge, app-server RPC, fleet client, and worker code.
- `fleet_manager/`: Python fleet manager service and MCP entrypoint.
- `scripts/`: deployment, repair, watchdog, Feishu provisioning, and OpenWrt
  patch helpers.
- `dist/`: compiled TypeScript output generated by `npm run build`.

## Basic Usage

Install dependencies and build the TypeScript bridge:

```sh
npm install
npm run build
```

Run the local bridge CLI:

```sh
npm run bridge
```

Run the Python fleet manager:

```sh
npm run fleet-manager
```

Run the fleet MCP server:

```sh
npm run fleet-mcp
```

Run checks:

```sh
npm run typecheck
npm test
npm run test:fleet
```

## Architecture

The project is split into three layers:

1. TypeScript bridge layer

   The `src/appserver`, `src/bridge`, `src/worker`, and `src/vscode-ipc`
   modules expose app-server RPC transports, bridge commands, worker
   protocols, and VS Code IPC helpers. The CLI entrypoint is `src/cli.ts`, and
   the packaged command is `codex-vscode-bridge`.

2. Fleet manager layer

   `fleet_manager/codex_fleet_manager.py` tracks sessions, tasks, chat
   bindings, and events. `fleet_manager/codex_fleet_mcp.mjs` exposes fleet
   operations through MCP so external agents can query or control the fleet.

3. OpenWrt integration layer

   Scripts under `scripts/patch_openwrt_*.py` patch the deployed OpenWrt bridge
   in place. They look for known deployment paths, write timestamped backups,
   and replace bounded function blocks in `codex_bridge_server.py`.

## Feishu Progress Card Patches

Two patch scripts maintain the OpenWrt Feishu progress-card behavior:

- `scripts/patch_openwrt_feishu_progress_edit.py`
- `scripts/patch_openwrt_feishu_session_progress.py`

Apply them on the OpenWrt host that contains the bridge deployment:

```sh
python3 scripts/patch_openwrt_feishu_progress_edit.py
python3 scripts/patch_openwrt_feishu_session_progress.py
```

Each script checks a small list of known target paths and patches every
existing target. Before writing, it creates a backup next to the target file
with a timestamped suffix such as `.bak-feishu-session-progress-YYYYmmddHHMMSS`.

### Behavior

`patch_openwrt_feishu_progress_edit.py` updates fleet-task completion watching
so Feishu group chats receive one editable interactive card per task. During
execution, the card is created once and updated as progress events arrive. On
completion or failure, the same card is updated to a green or red final state.

`patch_openwrt_feishu_session_progress.py` mirrors VS Code session events into
Feishu group chats. Assistant progress is rendered into an editable progress
card, user messages clear the active progress card and are sent as normal
messages, and final events mark the card complete. Task-scoped events update
the task's progress card when possible and fall back to sending a new final
card when the original message id cannot be found.

### Important Environment Variables

- `CODEX_FEISHU_PROGRESS_CARD_LIMIT`: maximum live progress text kept in a
  card. Default: `3000`.
- `CODEX_FEISHU_FINAL_CARD_CHUNK_SIZE`: markdown chunk size for final cards.
  Default: `2800`, clamped between `500` and `3000`.
- `CODEX_FEISHU_PROGRESS_SAFE_FINAL_LIMIT`: fallback final text length when
  Feishu rejects a full final card. Default: `700`.
- `CODEX_FEISHU_PROGRESS_EDIT_INTERVAL`: minimum interval between live card
  edits. Default: `1` second.
- `CODEX_FEISHU_PROGRESS_MIN_CHARS`: minimum live progress size before the
  first card is sent. Default: `1`.
- `CODEX_FEISHU_PROGRESS_BOOTSTRAP`: set to `0` to disable the initial
  "正在处理..." bootstrap card. Default: enabled.
- `CODEX_FEISHU_PROGRESS_BOOTSTRAP_TEXT`: custom bootstrap card text.
- `CODEX_FLEET_COMPLETION_PUSH_TIMEOUT`: task completion watcher timeout.
  Default: `900` seconds.
- `CODEX_FLEET_COMPLETION_PUSH_INTERVAL`: task polling interval. Default:
  `0.25` seconds.
- `CODEX_FLEET_SESSION_MIRROR_INTERVAL`: session mirror polling interval.
  Default: `0.5` seconds.
- `CODEX_FLEET_SESSION_MIRROR_TAIL`: number of recent fleet events to inspect
  per poll. Default: `100`.
- `CODEX_FLEET_SESSION_MIRROR_BINDING_TTL`: chat-binding refresh TTL.
  Default: `20` seconds.
- `CODEX_FEISHU_TASK_FINAL_ECHO_TTL`: window used to suppress duplicated final
  echoes between task and session event streams. Default: `45` seconds.

## Implementation Notes

The Feishu card logic uses Feishu interactive messages directly through
`/im/v1/messages` and message updates through `/im/v1/messages/{message_id}`.
The receive id type is selected from the target id: `oc_` values are treated as
group `chat_id`s, and other values are treated as `open_id`s.

Live progress is intentionally compacted: non-final cards keep only the latest
tail of progress text so repeated updates remain small. Final cards preserve
the full available final message by splitting it into multiple markdown
elements. If Feishu rejects a full final card or an update fails, the code
retries with a shortened safe final message and records the fallback path.

Progress state is persisted through the bridge state directory logs:

- `feishu-progress-card.log`: task progress-card send and update records.
- `feishu-session-mirror.log`: session mirror routing and skip decisions.
- `feishu-session-progress-mirror.log`: session progress-card records.
- `feishu-task-progress-fallback.log`: task final-card fallback records.

Duplicate suppression is stateful. Session final keys and recent task-final
fingerprints prevent the same final text from being sent twice when task events
and VS Code final events both describe the same completion.
